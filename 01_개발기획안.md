# 한성대 강의실 예약 챗봇 시스템 — 개발 기획안

## 1. 배경 및 목적
- 현재 강의실 마스터(153실, 24개 타입)와 학기별 사용률 데이터는 엑셀로 관리되고 있음.
- 학기 시작 전 정규 강의실 신청, 학기 중 임시 사용 신청, 충돌 조율을 사람이 수작업으로 처리할 경우 시간이 많이 들고 데드락이 잦음.
- 챗봇 인터페이스로 신청을 받고, 자동 배정 + 충돌 해소 + 메일 조율을 자동화하여 담당자/신청자 양쪽의 부담을 줄인다.

## 2. 데이터 분석 요약 (현재 폴더 기준)
- `강의실 데이터.xlsx`: `강의실코드`, `강의실명`, `강의실 구분`, `수용인원` (153행)
- `2025학년도 강의실 사용률.xlsx`: 학기별 시트(2025-1, 2025-2), 강의실별 요일/시간 사용률
- 강의실 타입 예: 계단식 강의실, 일반강의실, PC실습실(다양한 규모), 실기실/실습실, 제조공학실험실, 디자인개발실, 염색실 등
- 수용인원 분포: 4~145명, 가장 흔한 구간은 20·30·40·60명대

## 3. 사용자 시나리오
### A. 정규 학기 신청 (학기 시작 전, 신청 윈도우 오픈)
1. 챗봇 접속 → "정규 신청" 선택
2. 챗봇이 순서대로 질문:
   - 신청자 이름 / 이메일 / 소속(학과·동아리·부서)
   - 강의/행사 명
   - 강의실 종류 (예: PC실습실, 일반강의실, 계단식 등 — 마스터 타입에서 선택)
   - 필요 수용 인원
   - 사용 요일 (월~토 다중 선택)
   - 사용 시간대 (교시 또는 HH:MM-HH:MM)
   - 사용 주차 범위 (한 학기 전체 / 특정 주차)
   - 비고(필요 장비, 좌석 배치 등)
3. 신청 ID 발급 + 신청자 이메일로 접수 확인 발송

### B. 배정 단계 (신청 마감 직후, 일괄 실행)
1. 신청들을 (요일, 시간대) 슬롯 단위로 묶음
2. 슬롯별로 강의실 타입별 그룹화
3. 1차 매칭: 타입 일치 + 수용인원 ≥ 신청인원 + 해당 슬롯 공실 → 배정
4. 동일 강의실을 여러 신청이 원할 경우 → 같은 타입의 가용 강의실 풀에서 랜덤 셔플하여 분산 배정
5. 데드락(해당 타입 전부 점유) → 어드민이 "안내 메일 발송" 버튼으로 신청자 + 담당 사무실에 조율 메일(시뮬레이션)을 보낸 것으로 처리, 상태=`deadlock`으로 마킹, 재신청 가능

> **신청 우선순위(확정)**: 도착순(FCFS). `created_at` 오름차순. 동일 시각이면 신청 ID 오름차순.

### C. 임시 사용 신청 (학기 중, 일회성)
1. 챗봇 "임시 사용" 모드
2. 질문: 이용 목적, 희망 강의실 종류(또는 구체 강의실), 인원 규모, 시작/종료 일시
3. 시스템이 정규배정 DB + 임시예약 DB를 조회 → 해당 구간 공실 후보 제시
4. 사용자 선택 → 임시 예약 확정
5. 종료 시각 도래 시 스케줄러가 자동으로 상태=`expired` 처리 → 해당 슬롯 다시 가용 풀로 복귀

### D. 학기 전환
- 현재 학기 상태를 스냅샷(아카이브 테이블 또는 별도 SQLite 파일)으로 저장
- 신규 학기 인스턴스 생성 + 신청 윈도우 오픈
- 강의실 마스터는 학기와 독립적으로 유지(엑셀 변경 시 재시드)

## 4. 시스템 구성
```
[챗봇 UI (Streamlit)]
        │
        ▼
[대화 상태 머신 / 의도 분류]
        │
        ▼
[애플리케이션 서비스 계층]
   ├─ 신청 접수
   ├─ 배정 엔진 (충돌·랜덤·데드락)
   ├─ 임시 예약 관리
   └─ 학기 라이프사이클
        │
        ▼
[데이터 계층: SQLite]   ←─ 시드: 강의실 데이터.xlsx
        │
        ▼
[메일 발송 (SMTP)]  +  [스케줄러 (APScheduler)]
```

### 기술 스택
- 언어/런타임: Python 3.11
- 챗봇 UI: Streamlit (학내망 배포 용이)
- 대화 흐름: 룰 기반 상태머신 (선택지·유효성 검증 중심). 향후 Claude API로 자연어 이해 확장 옵션
- DB: SQLite (학기별 파일 분리도 가능). 동시성·규모 커지면 PostgreSQL로 이관 고려
- ORM: SQLAlchemy
- 메일: `smtplib` + 학내 SMTP 계정 (보내는 사람 = 강의실 운영 사무실)
- 스케줄러: APScheduler (임시예약 만료, 신청 마감 자동화)
- 데이터 시드: pandas

## 5. 데이터베이스 스키마 (초안)
```sql
classrooms        -- 마스터 (엑셀에서 시드)
  id PK, code, name, type, capacity, building, floor, note

semesters
  id PK, year, term, start_date, end_date,
  application_open_at, application_deadline_at,   -- 어드민에서 설정
  assignment_publish_at,                          -- 어드민에서 설정
  status ENUM(draft|open|closed|assigning|published|active|archived)

applications      -- 정규 학기 신청
  id PK, semester_id FK→semesters, applicant_name, email, affiliation,
  course_name, requested_type, capacity_needed,
  days JSON, time_slot, weeks JSON, notes,
  status ENUM(pending|assigned|deadlock|withdrawn),
  created_at

assignments       -- 정규 배정 결과
  id PK, application_id FK, classroom_id FK, semester_id FK,
  days JSON, time_slot, weeks JSON,
  method ENUM(auto|random|manual), assigned_at

temp_reservations -- 임시 사용
  id PK, classroom_id FK, requester_email, purpose, capacity,
  start_at, end_at,
  status ENUM(active|expired|cancelled), created_at

coordination_logs -- 데드락/조율 메일 기록
  id PK, application_id FK, kind ENUM(deadlock|conflict),
  recipients JSON, message TEXT, sent_at,
  resolution_status ENUM(pending|resolved)

mail_logs         -- 메일 시뮬레이션 로그 (프로토타입)
  id PK, event_kind ENUM(confirmation|assigned|deadlock|temp_confirmed|temp_expired),
  related_id INT,                  -- application_id 또는 temp_reservation_id
  recipients JSON, subject, body,
  simulated BOOL DEFAULT true, sent_at, triggered_by_admin
```

## 6. 배정 알고리즘 (의사코드)
```python
# 우선순위: 도착순(FCFS). created_at 오름차순, 동률이면 application_id 오름차순.
for slot in time_slots:                       # (요일, 시간대) 단위
    apps_in_slot = applications.filter(slot).order_by("created_at", "id")
    for room_type, apps in group_by(apps_in_slot, "requested_type"):
        rooms = classrooms.filter(
            type=room_type,
            capacity__gte=min_required(apps),
            free_at=slot,
        )
        random.shuffle(rooms)                 # 충돌 분산용 tie-breaker
        # apps는 이미 도착순. 정렬 변경 없음.
        for app in apps:                      # 먼저 도착한 신청이 먼저 자리 선택
            room = pick_smallest_fit(rooms, app.capacity_needed)
            if room:
                assign(app, room, slot, method="auto" or "random")
                rooms.remove(room)
            else:
                mark_deadlock(app)
                queue_mail_event(kind="deadlock", app=app)

# 메일은 어드민이 어드민 페이지에서 일괄/개별로 "보내기" 클릭 시 시뮬레이션 전송
```
- **도착순(FCFS)**: 먼저 신청한 사람이 먼저 자리 선택. 충돌 시 무작위 셔플은 어떤 강의실을 받을지에만 적용.
- **best-fit**: 후보 강의실 중 수용인원이 필요인원에 가장 가까운(작은 것부터) 강의실 선택 → 큰 강의실의 여유 보존
- 데드락은 즉시 종료 처리하지 않고 `deadlock` 상태로 둬, 조율 후 재신청 또는 수동 배정 가능

## 7. 메일 발송 정책 (프로토타입은 모킹)
- 프로토타입 단계에서는 **실제 SMTP 발송을 하지 않는다**. 모든 메일은 어드민 화면에서 트리거되는 시뮬레이션이다.
- 흐름:
  1. 시스템이 발송 대상 이벤트(접수 확인 / 배정 완료 / 데드락 / 임시예약 확정·만료)를 큐에 적재
  2. 어드민 페이지에 "안내 메일을 보내겠습니까?" 모달/버튼 노출
  3. "예" 클릭 → 메일 본문 미리보기 + `mail_logs` 테이블에 `sent_at`, `recipients`, `subject`, `body`, `simulated=true` 기록
  4. UI에는 "메일을 발송했습니다(시뮬레이션)" 토스트
- 실제 SMTP는 향후 `mailer.send_real()`로 스위치만 바꾸면 동작하도록 인터페이스 분리.
- 이벤트 종류
  - 신청 접수 확인 (신청자)
  - 정규 배정 통보 (신청자)
  - 데드락 조율 안내 (신청자 + 담당 사무실, 동일 타입 가용성 요약 + 재신청 안내)
  - 임시 예약 확정 / 만료 (신청자)

## 10. 결정사항 / 가정 / 남은 질문

### 결정됨
- **신청 우선순위**: 도착순(FCFS)
- **메일 발송**: 프로토타입은 모킹. 어드민 페이지에서 "안내 메일을 보내겠습니까?" 확인 후 시뮬레이션 발송 처리
- **마감일 / 배정 공개일**: 학기별로 어드민 페이지에서 직접 설정 (필드 = `application_deadline_at`, `assignment_publish_at`)

### 가정
- 시간 슬롯 단위는 한성대 학사 시간표 교시 기준 (1교시=09:00~09:50 등)으로 통일
- 학기 마감일이 지나면 시스템이 자동으로 신청 윈도우를 잠그고 배정 단계로 진입
- 배정 공개일 이전에는 신청자가 결과를 조회할 수 없음(어드민만 확인 가능)

### 남은 질문
- 임시 예약 가능 시간 범위 (예: 최소 30분, 최대 7일?)
- 강의실 마스터에 '관리소속' 정보(`사용률.xlsx`에는 있음)를 함께 시드할지
- 한 신청자가 동일 학기에 제출 가능한 최대 신청 수
- 데드락 메일 발송 단위 — 신청자별 개별 / 학기 전체 일괄 중 어느 쪽을 어드민이 선택 가능하게 할지
