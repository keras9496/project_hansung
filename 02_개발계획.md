# 한성대 강의실 예약 챗봇 시스템 — 개발 계획

전체 7주 일정(1인 풀타임 기준 추정). 각 Phase 끝에는 시연 가능한 산출물이 나오도록 설계.

## 확정 정책 (2026-05-14 기준)
- 신청 우선순위 = **도착순(FCFS)**
- 메일은 **실제 발송하지 않고** 어드민 페이지에서 시뮬레이션 — "안내 메일을 보내겠습니까?" 확인 → "발송 완료(시뮬레이션)" 표시 + `mail_logs` 기록
- 학기별 **신청 마감일·배정 공개일은 어드민에서 직접 설정** (`semesters.application_deadline_at`, `assignment_publish_at`)

## Phase 0 — 셋업 (1주차)
**목표**: 코드베이스 골격 + 강의실 데이터가 DB에 들어가 있는 상태.

작업
1. Python 3.11 가상환경, `requirements.txt`(streamlit, sqlalchemy, pandas, openpyxl, apscheduler, python-dotenv)
2. SQLAlchemy 모델 정의 (`classrooms`, `semesters` 우선)
3. `scripts/seed_classrooms.py`: 엑셀 → DB
4. `app/db.py`, `app/config.py`, `.env.example`

검수 기준
- `python scripts/seed_classrooms.py` 실행 시 153개 강의실이 DB에 적재됨
- `streamlit run app/main.py`로 빈 챗 화면이 뜸

## Phase 1 — 정규 신청 챗봇 (2~3주차)
**목표**: 신청자가 챗봇과 대화하여 신청을 끝내고 DB에 저장 + 접수 메일이 도착.

작업
1. 대화 상태 머신 설계 (질문 순서, 유효성 검증, 백트래킹 허용)
2. 정규 신청 플로우 구현 — 강의실 타입은 마스터에서 동적 셀렉터로 노출
3. 시간 슬롯 표준화 (교시 ↔ HH:MM 매핑 테이블)
4. `applications` 테이블 정의 및 저장
5. `mailer.py`로 접수 확인 메일 **이벤트 큐 적재** (실제 발송 안 함, `mail_logs`에 대기 상태로 기록)
6. 신청자가 자신의 신청 목록을 조회/철회할 수 있는 간단한 페이지

검수 기준
- 챗 한 번 완주하면 DB에 row 1개 + 메일 1통
- 잘못된 입력(이메일 형식, 인원 음수 등) 시 챗봇이 친절히 재질문

## Phase 2 — 배정 엔진 (4주차)
**목표**: 신청들이 자동으로 강의실에 배정되고, 충돌·데드락이 정상 처리됨.

작업
1. 슬롯 단위 그룹화 로직
2. best-fit + 랜덤 셔플 배정 알고리즘
3. 데드락 마킹 + 조율 메일 큐잉
4. `assignments` 테이블 적재, `applications.status` 갱신
5. `scripts/run_assignment.py`: 학기·슬롯 옵션으로 일괄 실행
6. 배정 통보 메일 / 데드락 조율 메일 템플릿 작성 (신청자/사무실 2종)

검수 기준
- 단위 테스트
  - 케이스1: 1신청 1강의실 → auto 배정
  - 케이스2: 동일 타입 3신청 vs 동일 타입 5강의실 → 모두 배정, 랜덤 분산 확인
  - 케이스3: 동일 타입 5신청 vs 동일 타입 3강의실 → 3 배정 + 2 데드락 + 메일 큐 2건
- 데드락 메일에 가용 강의실 요약 + 재신청 안내 링크가 포함

## Phase 3 — 임시 예약 (5주차)
**목표**: 학기 진행 중 임시 사용 신청이 챗봇으로 처리되고, 만료 시 자동 해제.

작업
1. 임시 예약 챗봇 플로우 (`flows_temp.py`)
2. 가용성 조회 로직 — 정규 배정 + 다른 임시 예약과 겹치지 않는 후보 산출
3. `temp_reservations` 테이블 운영
4. APScheduler로 만료 잡 (1분 주기 `status=expired` 처리)
5. 확정/만료 알림 메일

검수 기준
- 정규 배정과 시간 겹치면 후보에서 빠짐
- end_at 도래 시 1분 이내 만료 처리되는지 통합 테스트
- 임시 예약은 본인 이메일로 발급된 토큰으로만 취소 가능

## Phase 4 — 관리자 & 학기 전환 (6주차)
**목표**: 운영자가 현황을 보고, 학기 일정과 메일을 통제하며, 학기를 마감/개시할 수 있음.

작업
1. 관리자 페이지 (간단 토큰 인증): 신청·배정·데드락·임시예약 리스트, CSV 내보내기
2. **학기 설정 페이지**: `application_open_at`, `application_deadline_at`, `assignment_publish_at` 편집. 저장 시 상태 자동 전이 (예: 지금 시각이 deadline 이후이면 `closed`로 전환)
3. **메일 발송함 페이지**: 큐에 쌓인 메일 이벤트 목록 → 본문 미리보기 → "안내 메일을 보내겠습니까?" 확인 → `simulated=true`로 `mail_logs`에 기록, UI에는 "발송 완료(시뮬레이션)" 표시. 일괄 발송과 개별 발송 모두 지원
4. 수동 배정/재배정 UI (데드락 해결용)
5. `scripts/close_semester.py`: 현재 학기 아카이브, 임시 예약 정리
6. `scripts/start_semester.py`: 새 학기 인스턴스 + 신청 윈도우 오픈
7. 사용률 리포트: 배정 결과를 기존 사용률 엑셀과 같은 포맷으로 출력

검수 기준
- 학기를 마감→개시했을 때 신규 신청이 새 학기에 귀속
- 아카이브된 학기 데이터가 조회 가능

## Phase 5 — 테스트 & 배포 (7주차)
작업
1. 시나리오 통합 테스트 (정규→배정→데드락→재신청→임시예약→학기마감 풀 시나리오)
2. 부하 가정: 한 학기 최대 신청 ~1000건에 배정 < 30초
3. 운영 가이드 (`README.md`) — 메일 자격증명 설정, 학기 운영 절차, 장애 시 롤백
4. 학내 서버 배포 (Windows 서비스 또는 Linux systemd 유닛)
5. 백업 정책 — SQLite 파일 일일 백업

## 리스크와 완화
| 리스크 | 영향 | 완화 |
|---|---|---|
| 시간 슬롯 표준이 학사일정과 어긋남 | 배정 오류 | Phase 1에서 학사 시간표 확정 후 매핑 테이블 고정 |
| 학교 SMTP 정책 변경 / SPF·DKIM 미설정 | 메일 미수신 | Phase 1 초반에 발신 테스트, 필요 시 SendGrid 등 대체 |
| 강의실 마스터 변경(증/감) | 시드 충돌 | 시드 스크립트는 upsert(code 기준), 변경 이력은 별도 테이블 |
| 데드락 메일 폭주 | 사무실 부담 | 학기당 1회 배치 메일 + 다이제스트 옵션 |
| 동시 임시예약 경합 | 더블 부킹 | DB 트랜잭션 + 가용성 검증을 같은 트랜잭션에 포함 |

## 의사결정 항목 — 진행 상태
| # | 항목 | 상태 |
|---|---|---|
| 1 | 신청 우선순위 | ✅ 도착순(FCFS) |
| 2 | 신청 마감일 / 배정 공개일 | ✅ 어드민 페이지에서 학기별 설정 |
| 3 | 메일 발신 | ✅ 프로토타입은 시뮬레이션(실제 발송 X) |
| 4 | 임시 예약 허용 범위 | ⏳ 미정 — 기본값 제안: 최소 30분, 최대 7일, 14일 전부터 신청 가능 |
| 5 | 관리자 인증 | ⏳ 미정 — 프로토타입은 환경변수 토큰 1개로 단순 처리 제안 |
| 6 | 강의실 마스터 '관리소속' 시드 | ⏳ 미정 — 일단 `사용률.xlsx`에서 가져와 컬럼으로 보관 권장 |
| 7 | 배포 환경 | ⏳ 미정 — 프로토타입은 로컬/Streamlit |

## 즉시 다음 액션
- Phase 0 코드 골격은 본 문서 갱신과 함께 작성 완료(`app/`, `scripts/seed_classrooms.py`, `requirements.txt`).
- 사용자가 가상환경을 만들고 시드를 실행하면 Phase 1 착수 가능:
  ```bash
  python -m venv .venv
  .venv\Scripts\activate           # Windows
  pip install -r requirements.txt
  python scripts/seed_classrooms.py
  streamlit run app/main.py
  ```
